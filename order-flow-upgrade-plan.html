The Problem

Our bubbles aggregate all trades in a 250ms window into one VWAP-averaged point. This answers "is the market buying or selling?" but can't answer the questions that actually drive trading decisions:

  • Where exactly is buying/selling concentrated? (per-price-level volume)
  • How aggressively? (bid vs ask imbalance at each level)
  • Is it working? (absorption — big volume but no price movement)
  • Is it institutional? (stacked imbalances, large prints, icebergs)

The Three-Phase Upgrade

PHASE 1 Overlay Markers on Current Bubbles

Keep the flowing trail. Add absorption markers, large print highlights, delta divergence arrows, and volume-at-price clusters as visual overlays. No data model change.

Effort: ~2-3 hours  |  Patterns enabled: 3 of 6

PHASE 2 Per-Price-Level Bubble Aggregation

Change aggregation from "one VWAP bubble per time slice" to "one bubble per price level per time slice." Each bubble sits at its exact traded price. Unlocks imbalance detection and proper volume profiling.

Effort: ~4-6 hours  |  Patterns enabled: 5 of 6

PHASE 3 Footprint Chart View (Optional)

Add a dedicated footprint chart panel showing bid×ask numbers at each price level within each time bar. Stacked imbalances highlighted. The gold standard for institutional flow reading.

Effort: ~6-8 hours  |  Patterns enabled: 6 of 6

Data We Have (Alpaca SIP + ThetaData)

Our Alpaca Algo Trader Plus subscription gives us everything needed:

  • Every trade tick: price, size, exchange, conditions, timestamp
  • NBBO quotes: best bid/ask with sizes — lets us classify trades as aggressive buy (at/above ask) vs aggressive sell (at/below bid)
  • What we DON'T have: Full Level 2 order book depth — so no Bookmap-style resting liquidity heatmap. But we have enough for everything else.

Current Bubble System

CURRENT: VWAP-Averaged Flowing Bubbles CURRENT

How It Works Now

Every 250ms, all trades in that window are VWAP-averaged into a single point. The bubble's Y = volume-weighted average price, size = total volume (percentile-scaled), color = net delta ratio (green=buy, red=sell).

What a Pro Trader Sees

Momentum Direction ✓

Green trail flowing right = buyers in control. Red = sellers. This works.

Volume Spikes ✓

Bigger bubbles = more volume. Relative sizing via percentiles works across market conditions.

Price Level Detail ✗

All trades averaged into one point. Can't see WHERE volume concentrated. A 10,000 share trade at $570.25 mixed with 500 shares at $570.50 appears as one bubble at ~$570.26.

Imbalance Detection ✗

Can't compare buy volume at price X vs sell volume at price X-1. The diagonal imbalance — the #1 institutional signature — is invisible.

Absorption ✗ (visual)

Detected server-side in confluence engine but not shown on chart. Trader can't see it.

New Capabilities

▓▓▓ Stacked Imbalance Detection

With per-price data, we can compare aggressive buy volume at price X vs aggressive sell volume at price X-0.01 (diagonal comparison). When this ratio is 3:1+ across 3 or more consecutive price levels, that's a stacked imbalance — the strongest institutional signal in order flow.

Visually: highlight those price levels with a bright vertical bar or connected markers.

MEDIUM Requires new aggregation logic + NBBO classification at ingest + imbalance comparison algorithm. ~200 lines.

⦿ Exhaustion at Price Levels

When the highest price bubble in a time bar has large buy volume but the next bar's highest price is lower — that's buying exhaustion at the top. The bubbles make this visible because you can see the big green dot at the peak with no follow-through.

EASY Falls out naturally from per-price rendering. No extra code needed.

│ Precise Volume Profile

The right-side price ladder becomes truly accurate — each level shows volume that actually traded there, not approximated from VWAP buckets.

EASY Direct mapping from per-price data. Actually simplifies ladder code.

Performance Consideration

More bubbles per time slice (3-8 vs 1). At 250ms aggregation with a 2-minute window, worst case goes from ~480 bubbles to ~2,000. PixiJS handles this easily (tested up to 10,000 sprites). The GPU renderer we already have was built for this.

Phase 3: Footprint Chart View (Optional Add-on)

A dedicated panel showing the classic bid×ask footprint — numeric values at each price level within each time bar. This is what Sierra Chart, NinjaTrader, and ATAS traders use.

PHASE 3: Bid × Ask Footprint Chart PHASE 3

What This Shows

Each Cell = One Price Level in One Time Bar

Left number (red): volume sold at bid (aggressive sellers)
Right number (green): volume bought at ask (aggressive buyers)
Background color intensity = total volume at that level
Highlighted cells: imbalance ratio ≥ 3:1

Delta Column (right side of each bar)

Net delta (buy - sell) at each price level. Green = net buyers, Red = net sellers. The pattern of deltas across price levels reveals whether buying is concentrated at lows (accumulation) or highs (exhaustion).

POC (Point of Control)

The price level with the highest total volume in each bar. Marked with a dot or bar. This is the "fair price" according to volume — price tends to return to POC.

When to Use Footprint vs Bubbles

Use Bubbles For:

Quick momentum reading, seeing overall flow direction, scanning multiple timeframes, visual pattern recognition at a glance

Use Footprint For:

Entry timing (reading exact levels), confirming institutional presence, identifying absorption/iceberg behavior, precise risk placement

Zero risk to existing functionality. Immediate visual value. Gives you absorption, large prints, and divergence on the chart right away. Takes 2-3 hours.

Then Phase 2

This is the critical upgrade. Per-price bubbles unlock stacked imbalances — the most valuable institutional signal. The visual style stays similar (flowing bubbles) but each bubble now means something actionable. The PixiJS renderer already handles the increased bubble count.

Phase 3 only if needed

Full footprint view is the professional gold standard but has a steep learning curve. If Phase 2 bubbles give you enough information, Phase 3 may not be necessary. It's a nice-to-have for deep tape reading.

Data Pipeline Changes

Phase 1 — Backend → Frontend Data Additions

Push existing server-side detections to the frontend via WebSocket:

  • absorption_levels: array of {price, bias, volume} from confluence engine
  • divergence_state: {type: "bullish"|"bearish", strength: 0-1}
  • Large print flag on individual ticks: is_large: true when size ≥ 5000

Phase 2 — Aggregation Model Change

Current: ticks → group by time bucket → VWAP average → one cell per bucket

New: ticks → group by (time bucket, rounded price) → per-level stats → N cells per bucket

Each cell gains: {price, aggBuyVol, aggSellVol, passiveBuyVol, passiveSellVol, tradeCount}

Trade classification uses NBBO: trade at/above ask = aggressive buy, at/below bid = aggressive sell.

Phase 3 — No Additional Backend Changes

Uses the same per-price data from Phase 2. Only new frontend rendering code.
